# 人工评分模块设计

日期：2026-05-07

## 背景

当前软件已经能对毫米波人体成像图片进行自动质量评估，但实际试用发现部分视觉观感较差的图片仍可能获得偏高分数。为了后续重构指标体系，需要先收集“图片-人工评分”对应关系，并支持多个看图员对同一数据集独立评分，再进行平均、加权或其他统计分析。

本设计在现有 FastAPI + React/Vite 应用中新增一个相对独立的人工评分模块。该模块与当前自动评估界面共享图片文件和图片 ID，但拥有独立入口、登录、任务、评分和导出流程。

## 目标

- 支持用户名 + 密码登录。
- 支持管理员和看图员两类角色。
- 支持管理员创建评分任务，并指定看图员参与。
- 支持一个数据集由多个看图员独立评分。
- 支持看图员对五个维度进行 `0-10` 分评分，步进 `0.5`。
- 支持盲评：看图员只看原图，不看自动评分、自动指标、AOI、伪影或条纹叠加层。
- 支持管理员查看任务进度、人工评分均值，并导出原始评分数据。
- 保持现有自动评估流程和 `data/state.json` 不被打散。

## 非目标

- 第一版不实现 PostgreSQL/MySQL 或网络级部署方案。
- 第一版不做复杂审计日志、评分锁定审批流或多级权限模型。
- 第一版不重构现有自动评分算法。
- 第一版不在看图员界面显示任何自动指标解释或算法提示。
- 第一版不直接训练或重构指标体系，只负责收集可用人工标签数据。

## 推荐方案

采用“同应用内独立模块入口 + SQLite 人工评分库”的方案。

前端新增独立的人工评分入口，例如顶部导航中的“人工评分”。进入该入口后，根据登录用户角色显示管理员工作台或看图员工作台。它不嵌入现有自动评估主面板，避免盲评流程受到自动指标界面影响。

后端新增人工评分相关 API 和一个本地 SQLite 文件 `data/manual_rating.db`。现有图片导入、图片文件、自动指标和 `image_id` 继续由现有 `ImageRepository` 管理；人工评分数据库通过 `image_id` 引用现有图片记录。

## 角色与权限

管理员：

- 创建、停用用户。
- 创建评分任务。
- 从现有图片记录中选择数据集或批次。
- 指定参与任务的看图员。
- 查看任务进度。
- 查看每张图片、每个维度的评分人数和均值。
- 导出评分数据。

看图员：

- 登录后查看分配给自己的任务。
- 对任务中的图片进行盲评。
- 保存和修改自己提交的评分。
- 查看自己的完成进度。

权限约束：

- 看图员不能访问自动评估指标、评分权重、AOI、mask 或 overlay API。
- 看图员不能查看其他看图员的评分。
- 看图员不能创建任务或管理用户。

## 数据模型

新增 SQLite 文件：`data/manual_rating.db`。

表结构：

### users

- `id`: 文本主键，UUID。
- `username`: 唯一用户名。
- `display_name`: 显示名称。
- `password_hash`: 密码哈希。
- `role`: `admin` 或 `reviewer`。
- `active`: 是否启用。
- `created_at`: 创建时间。

### datasets

- `id`: 文本主键，UUID。
- `name`: 数据集名称。
- `source`: 数据集来源，例如 `existing_images`。
- `experiment_group`: 可选，用于记录来自现有图片的实验组。
- `batch`: 可选，用于记录来自现有图片的批次。
- `created_by`: 管理员用户 ID。
- `created_at`: 创建时间。

### dataset_images

- `dataset_id`: 数据集 ID。
- `image_id`: 现有 `data/state.json` 中的图片 ID。
- `sort_order`: 任务内顺序。

唯一约束：`dataset_id + image_id`。

### rating_tasks

- `id`: 文本主键，UUID。
- `dataset_id`: 数据集 ID。
- `name`: 任务名称。
- `description`: 可选说明。
- `status`: `draft`、`active`、`closed`。
- `created_by`: 管理员用户 ID。
- `created_at`: 创建时间。

### task_reviewers

- `task_id`: 任务 ID。
- `reviewer_id`: 看图员用户 ID。
- `weight`: 可选权重，默认 `1.0`，第一版主要用于导出和后续统计。

唯一约束：`task_id + reviewer_id`。

### manual_ratings

- `id`: 文本主键，UUID。
- `task_id`: 任务 ID。
- `image_id`: 图片 ID。
- `reviewer_id`: 看图员用户 ID。
- `sharpness_score`: 清晰度，`0-10`，步进 `0.5`。
- `significance_score`: 显著性，`0-10`，步进 `0.5`。
- `artifact_suppression_score`: 伪影抑制，`0-10`，步进 `0.5`。
- `structure_score`: 结构完整性，`0-10`，步进 `0.5`。
- `detail_score`: 细节保真度，`0-10`，步进 `0.5`。
- `comment`: 可选备注。
- `created_at`: 首次提交时间。
- `updated_at`: 最近修改时间。

唯一约束：`task_id + image_id + reviewer_id`，用于保证同一个看图员对同一任务内同一张图片只有一条当前评分记录。

## 后端 API

建议新增 API 分组：

### 认证

- `POST /api/auth/login`: 用户名和密码登录，返回当前用户信息并设置 session 或令牌。
- `POST /api/auth/logout`: 退出登录。
- `GET /api/auth/me`: 返回当前登录用户。

第一版使用服务端 session cookie。后端在登录成功后设置 HttpOnly cookie，前端通过 `GET /api/auth/me` 恢复当前用户状态，避免在前端持久化敏感令牌。

### 用户管理

- `GET /api/manual/users`: 管理员列出用户。
- `POST /api/manual/users`: 管理员创建用户。
- `PATCH /api/manual/users/{user_id}`: 管理员更新显示名、角色、启用状态或重置密码。

### 数据集与任务

- `GET /api/manual/datasets`: 管理员列出人工评分数据集。
- `POST /api/manual/datasets`: 管理员从现有图片按批次、实验组或选择列表创建数据集。
- `GET /api/manual/tasks`: 管理员列出全部任务；看图员列出自己的任务。
- `POST /api/manual/tasks`: 管理员创建任务并分配看图员。
- `GET /api/manual/tasks/{task_id}`: 返回任务详情、进度和当前用户权限范围内的数据。
- `PATCH /api/manual/tasks/{task_id}`: 管理员更新任务状态或分配看图员。

### 评分

- `GET /api/manual/tasks/{task_id}/next`: 看图员获取下一张未评分图片。
- `GET /api/manual/tasks/{task_id}/images/{image_id}`: 看图员获取某张图片的盲评数据，只包含原图 URL、文件名、任务进度和本人已有评分。
- `PUT /api/manual/tasks/{task_id}/images/{image_id}/rating`: 看图员创建或更新评分。
- `GET /api/manual/tasks/{task_id}/summary`: 管理员获取任务统计。

### 导出

- `GET /api/manual/tasks/{task_id}/export/csv`
- `GET /api/manual/tasks/{task_id}/export/excel`

导出字段至少包括：任务 ID、任务名、数据集 ID、图片 ID、文件名、看图员 ID、看图员显示名、五维人工分、五维均值、看图员权重、提交时间、更新时间。

## 前端界面

### 登录页

未登录时显示登录页。登录成功后按角色进入对应工作台。

### 管理员工作台

主要区域：

- 任务列表：任务名、数据集、状态、看图员数量、完成率。
- 创建任务：选择现有实验组、批次或图片列表生成数据集，并指定看图员。
- 用户管理：创建用户、重置密码、停用用户。
- 任务详情：每个看图员完成进度、每张图片评分人数、各维度均值、导出按钮。

### 看图员工作台

主要区域：

- 我的任务列表：任务名、图片数量、已完成数量、状态。
- 盲评页面：只显示原图、文件名、当前进度、五个评分输入、备注、保存并下一张。
- 历史评分列表：允许回到已评分图片修改。

盲评页面禁止显示：

- 自动质量总分。
- 自动五维归一化指标。
- 原始物理指标。
- AOI/mask/overlay。
- 排名或其他看图员评分。

## 评分规则

五个维度沿用当前自动评分维度：

- `sharpness_score`: 清晰度。
- `significance_score`: 显著性。
- `artifact_suppression_score`: 伪影抑制。
- `structure_score`: 结构完整性。
- `detail_score`: 细节保真度。

每项范围为 `0-10`，步进 `0.5`。后端必须校验：

- 字段完整。
- 数值在 `0` 到 `10` 之间。
- 数值是 `0.5` 的整数倍。
- 当前用户已被分配到该任务。
- 任务状态为 `active`。

## 与现有系统的集成

现有 `ImageRepository` 继续作为图片事实来源。人工评分模块读取现有图片记录，并通过现有 `/uploads/{image_id}` 服务原图。

人工评分模块不得修改：

- `data/state.json` 的自动评估记录结构。
- 现有自动评分权重和排序逻辑。
- 现有导入和自动评分接口行为。

删除图片时采用保护策略：如果图片已被人工评分任务引用，则自动评估界面删除该图片时返回明确错误，提示先关闭或清理关联评分任务，避免人工评分记录悬空。

## 错误处理

- 未登录访问人工评分 API 返回 `401`。
- 权限不足返回 `403`。
- 任务、用户或图片不存在返回 `404`。
- 评分超出范围或非 `0.5` 步进返回 `422`。
- 看图员访问未分配任务返回 `403`。
- 关闭任务后提交评分返回 `409`。

## 测试策略

后端测试：

- 用户创建、密码登录、角色权限。
- 管理员创建数据集和评分任务。
- 看图员只能看到分配给自己的任务。
- 评分范围和 `0.5` 步进校验。
- 同一看图员重复提交同一图片时更新现有评分，而不是插入重复记录。
- 多看图员对同一图片评分后，管理员统计均值正确。
- 已被评分任务引用的图片删除保护。

前端测试：

- 未登录时显示登录页。
- 管理员看到任务、用户、导出入口。
- 看图员只看到自己的任务和盲评页面。
- 盲评页面不渲染自动评分、自动指标或 overlay 切换控件。
- 评分输入支持 `0.5` 步进并提交正确 payload。

手工验证：

- 创建两个看图员和一个管理员。
- 从现有导入图片创建一个数据集。
- 创建任务并分配两个看图员。
- 两个看图员分别提交同一批图片评分。
- 管理员查看完成率和均值。
- 导出 CSV/Excel 并确认包含原始评分和汇总字段。

## 实施顺序建议

1. 新增 SQLite 仓储层和数据表迁移初始化。
2. 新增认证和用户管理 API。
3. 新增数据集、任务和评分 API。
4. 新增前端登录和角色路由。
5. 新增管理员任务/用户工作台。
6. 新增看图员盲评工作台。
7. 新增统计和导出。
8. 补齐后端、前端和手工验证。

## 已确认决策

- 数据集来源支持复用现有图片库，也允许后续扩展单独导入；每个评分任务必须绑定明确数据集或批次。
- 登录采用用户名 + 密码。
- 管理员创建评分任务并指定看图员。
- 看图员只看原图，不看自动指标和叠加层。
- 模块形态采用同应用内独立入口。
- 人工评分数据使用 SQLite。
- 五维评分支持 `0.5` 步进。
